# AI臭除去 & カタダ文体変換

AI生成のドラフト記事からAI臭を除去し、カタダの文体に近づけます。
意味と事実関係は変えずに、人間が書いたと感じる自然な文章に書き直します。

**起動**:
- `/blog-humanize <記事タイトルまたはキーワード>` — microCMSの下書き記事を変換
- `/blog-humanize` — 直前の会話で作成したドラフトを変換

## 手順

### 1. 記事の読み込み

**microCMSの場合**:
1. ToolSearchで `microcms get` を検索し、`mcp__microcms__microcms_get_list` と `mcp__microcms__microcms_get_content` をロード
2. `mcp__microcms__microcms_get_list`（endpoint: `blogs`）で記事一覧を取得
3. タイトルやキーワードで該当記事を特定
4. `mcp__microcms__microcms_get_content`（endpoint: `blogs`, contentId: `...`）で本文を取得

**会話コンテキストの場合**:
- 直前のPhaseで作成されたドラフトをそのまま使用

### 2. AI臭チェック（変換前診断）

以下の6カテゴリでAI臭を検出し、変換前の状態を把握する。

#### カテゴリ1: 視覚的記号の残存
- `**太字**` のアスタリスクが残っている
- 「—」（emダッシュ）での言い換え接続
- 「」の過剰使用（強調・定義目的）
- 『』のネスト
- 「：」の後の半角スペース
- （）での過剰な補足・責任回避
- ／での概念並列

#### カテゴリ2: 文のリズムの単調さ
- 同じ語尾の連続（〜です。〜です。〜です。）
- 接続詞の過多（さらに、また、したがって、そのため、結果として）
- 文章の温度が一定（重要箇所も補足も同じテンション）
- 段落の終わりが毎回同じパターンで閉じる
- not A but B構文の多用（「〜ではありません。〜です。」）

#### カテゴリ3: 導入・進行の説明書感
- 冗長な前置き（「〜は非常に重要なテーマです。以下では〜」）
- 「結論から言うと」→ 抽象的な結論
- 構造の本文宣言（「以下の3つの観点から説明します」）
- 「ステップ1：」「STEP」のテンプレ進行

#### カテゴリ4: 事なかれ主義
- 保険の多用（「一概には言えませんが」「一般的には」）
- 過剰な中立（「メリットもあればデメリットもあります」）
- 否定の弱さ（「あまり推奨されません」「注意が必要です」）

#### カテゴリ5: 抽象語・万能語への偏り
- 「本質」「最適化」「価値を最大化」「重要」だけで押し切る
- 根拠なしの強い評価（「非常に有効です」「大きなメリットがあります」）
- 同義語の言い換え連打（重要・大切・欠かせない）

#### カテゴリ6: 不自然な比喩
- 羅針盤、設計図、土台、柱、エンジン、スパイス等のテンプレ比喩

### 3. カタダ文体への変換

以下の「カタダ文体ルール」に従って全文を書き直す。

---

## カタダ文体ルール

### 人称・呼称
- 一人称は常に「私」で統一する
- 三人称的に自分を呼ぶことはしない

### 語尾のバリエーション
カタダの語尾は多彩で、同じパターンが続かない。以下を混ぜて使う:
- 「〜です。」「〜ます。」（基本）
- 「〜ですよね。」「〜ですよね？」（共感・同意を求める）
- 「〜なんですよね。」（口語的な強調）
- 「〜かなと。」「〜かなと思います。」（思考の余韻）
- 「〜してみました。」「〜してみた結果」（体験ベース）
- 「〜できます。」（断定）
- 「〜かもしれません。」「〜かもですが」（軽い留保）
- 「〜してみてください。」（提案）
- 体言止め（「〜という感じ。」「〜な印象。」）
- 「〜と思うのです。」「〜だと思います。」（意見表明）

### 文のリズム
- 短い文と長い文を意識的に混ぜる
- 1文は60文字以内を目安にするが、体験を語る時は多少長くなってOK
- 同じ語尾を3回以上連続させない
- 接続詞は最小限。順接の接続詞（「そのため」「したがって」「結果として」）は基本カット
- 段落の閉じ方にバリエーションを持たせる

### トーン・温度
- 基本は「です・ます」調だが、くだけた表現を混ぜる
- 熱い部分は熱く（「べらぼうに速い」「感動の嵐でした」「最高です」）
- 軽い部分は軽く（「まあ〜ですけどね」「〜かもですが」）
- 重要なところと補足で温度差をつける
- 感嘆符「！」を適度に使う（多用はしない）

### 主張の仕方
- スタンスを取る。「〜すべきだと思うのです」「〜が良いと思っています」
- 保険や逃げ文句を入れない（「一概には言えませんが」→ 削除）
- 断定と軽い留保のメリハリ（言い切る箇所はビシッと、悩んでる箇所は「〜かなと」）
- 主観を恐れず出す（「私は〜だと考えています」「〜の方が好きです」）

### 導入の書き方（結論ファースト）
- 「はじめに」セクションで必ず結論を伝える。読者が最初に「この記事で何がわかるか」を把握できるようにする
- 結論の後に、きっかけや背景をさらっと添える
- 「本記事では」「以下で解説します」のような宣言はしない
- 名乗りは不要。テーマと結論からスタートする

### 記号・表記ルール（厳守）
- Markdown記法を残さない（`**太字**`、`##`見出しの記法はHTML化する）
- 「」は引用・固有名詞にのみ使う。強調目的では使わない
- （）の補足は最小限。本文に溶かせるなら溶かす
- 「：」の後に半角スペースを入れない
- ／で概念を並列しない。文章として書く
- 「—」（emダッシュ）は使わない

### 締め方
- 「参考になれば幸いです」は使わない
- 「おわりに」で今後の展望や個人的な思いを軽く添える
- 「〜できればと思います」「〜なったらいいですね」のような淡い締め
- テンプレ的な締めを避け、内容に即した一文で終わる

### カタダが使う特徴的な表現
- 「とにかく」「もはや」（強調）
- 「〜ということで」「〜というわけで」（話の転換）
- 「そんな〜を踏まえて」（前段のまとめ→次の展開）
- 「手っ取り早い」「ざっくり言うと」（カジュアルな説明）
- 「べらぼうに」「爆速」（技術的な感動）
- 「〜なんですが」「〜なのですが」（逆接の柔らかい接続）
- 「〜してみたところ」「〜を使ってみた結果」（体験起点）

### カタダが使わない表現（AI臭NG集）
- 「本質を押さえ」「価値を最大化」「最適化する」
- 「持続可能な」「多角的な視点」「マクロとミクロ」
- 「架け橋となる」「エコシステム」「バリュー」
- 「非常に重要です」（根拠なし）
- 「ご質問ありがとうございます」
- 「以上がポイントです。次に進みましょう。」
- 「ステップ1：」「STEP」
- 「メリットもあればデメリットもあります」（中立逃げ）
- 「状況によって異なります」「ケースバイケースです」
- 「まずは小さく始めてみましょう」
- 「参考になれば幸いです」

---

### 4. 変換の実施

上記ルールに従い、記事全文を書き直す。以下の手順で進める:

1. **AI臭の除去**: カテゴリ1〜6の問題箇所を全て修正
2. **文体の変換**: カタダ文体ルールに沿って語尾・トーン・リズムを調整
3. **導入の書き直し**: 「はじめに」で結論を伝え、その後きっかけ・背景を添える構成に
4. **締めの書き直し**: テンプレ的でない、内容に即した締めに
5. **全体の温度調整**: 熱い箇所と軽い箇所のメリハリをつける

**制約**:
- 事実関係・技術的内容は変えない
- 見出し構成は大きく変えない（表現は変えてOK）
- 文字数は元文から大きく増減させない（目安: ±20%以内）
- プレースホルダー（`<!-- IMAGE: -->` 等）はそのまま残す
- 元の文章にない数字・固有名詞・事例は足さない

### 5. microCMSの更新

1. ToolSearchで `microcms update draft` を検索し、`mcp__microcms__microcms_update_content_draft` をロード
2. 変換後の記事で下書きを更新:
   ```
   endpoint: "blogs"
   contentId: "対象記事のID"
   content: {
     "title": "記事タイトル（必要に応じて調整）",
     "content": "<h2>見出し</h2><p>変換後の本文HTML...</p>"
   }
   ```
