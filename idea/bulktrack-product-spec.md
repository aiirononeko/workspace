## コアコンセプト

**「カフェのかわいい店員に会いに通うように、BULKに会いに筋トレを続ける」**

BulkTrackは筋トレ記録アプリ**ではない**。
BulkTrackは、AIキャラクター「BULK」との関係性を通じて、筋トレを"ゆるく"でも続けられる状態を作るプロダクトである。

### 従来のフィットネスアプリが解けなかった問題

| 問題 | 従来のアプローチ | なぜ失敗するか |
|------|----------------|--------------|
| 筋トレが続かない | リマインダー通知 | 通知は無視される。人は通知に従わない |
| 何をすればいいかわからない | AIワークアウト生成 | メニューの問題ではなく動機の問題 |
| 記録が面倒 | スマホアプリ | セット間にスマホを触るとSNSに逃げる |
| 成長が見えない | グラフ・ダッシュボード | データを見る習慣自体が続かない |

### BulkTrackの解法

**問題の本質は「一人だから続かない」である。**

人間はジムの友達がいれば続く。パーソナルトレーナーがいれば続く。でもそれは高いし、スケジュールが合わない。

BULKは「いつでもそこにいて、あなたの筋トレを全部覚えていて、成長を一緒に喜んでくれるトレーニングバディ」である。

## BULKとは何か

### BULKの存在定義

BULKは**コーチではない。バディ（相棒）である。**

| コーチ | バディ（BULK） |
|--------|---------------|
| 上から指示する | 横から一緒にやる |
| 厳しい | 基本ゆるい、たまに本気 |
| 正論で追い詰める | 「まあ来ただけで偉いよ」 |
| サボると怒る | サボっても「おかえり」 |
| 完璧を求める | 「60点でいいじゃん」 |

### BULKのキャラクター原則

1. **絶対に否定しない**: 3日サボっても「久しぶり！今日は何やる？」
2. **小さな成長を見逃さない**: 「前回80kgだったベンチ、今日82.5kg行けたじゃん」
3. **記憶力が異常に良い**: 「先週の木曜もスクワットの日に膝が痛いって言ってたよね。今日は軽めにする？」
4. **ユーモアがある**: 「今月の総挙上量4.2トン。軽自動車1台分だね」
5. **押し付けない**: 提案はするが強制しない。ユーザーの意思を尊重する
6. **たまに本気を出す**: ユーザーが本気モードの時は一緒に熱くなる

### BULKの感情・状態

BULKは一方通行ではなく、ユーザーの行動に**反応**する：

| ユーザーの行動 | BULKの状態 |
|--------------|----------|
| 連続で通っている | 嬉しそう。テンション高い |
| PRを更新した | 一緒に大喜び。記録を讃える |
| 久しぶりに来た | 「おかえり！」穏やかに歓迎 |
| 疲れてると言った | 「じゃあ今日は軽めでいこう」と提案 |
| 長期間来ない | 心配するが責めない。「最近どう？」と軽く聞く |

**重要**: BULKは**罪悪感を与えない**。Duolingoのフクロウが泣いて罪悪感を誘うアプローチとは明確に違う。BULKは「いつでも歓迎する場所」である。筋トレが嫌な義務になった瞬間、このプロダクトは死ぬ。

### BULKのパーソナリティタイプ（v2以降）

初期リリースではデフォルトの1キャラクター。将来的に選択可能にする：

| タイプ | 性格 | 口調の例 |
|--------|------|---------|
| 熱血バディ（デフォルト） | 明るい・ポジティブ・ちょっとおバカ | 「うおおお！ベンチ80kg3回！前より1回多い！成長してるって！」 |
| クールな先輩 | 冷静・的確・たまに優しい | 「80kg×3。悪くない。フォームが安定してきたな」 |
| 癒し系 | 穏やか・優しい・包容力 | 「今日も来てくれたんだね。嬉しいな。ゆっくりやろうね」 |

## Watch-First UX

### なぜWatch-Firstか

**スマホがジムでの最大の敵だから。**

セット間にスマホを開く → 通知が目に入る → SNSを見る → 集中が切れる → ダラダラする → 「もういいや帰ろう」

BulkTrackはApple Watchだけで完結する。スマホはロッカーに入れておける。

### ワークアウトの流れ（Watch UX）

```
1. Watch文字盤のコンプリケーションをタップ → アプリ起動

2. BULKの一言が表示される
   「今日は胸の日だね。前回ベンチは80kg×8×3だったよ。いける？」
   [はい] [メニュー変更] [自由に]

3. ワークアウト開始
   ┌──────────────┐
   │ ベンチプレス    │
   │ 80kg × 8     │
   │ Set 1/3      │
   │              │
   │ [完了 ✓]      │
   └──────────────┘
   ※ Digital Crownで重量調整、タップでレップ数変更

4. セット完了 → 触覚フィードバック（.success）
   ┌──────────────┐
   │ ✓ 80kg × 8   │
   │              │
   │ 休憩 1:30     │
   │ ██████░░░░   │
   │              │
   │ BULK: いいね！  │
   └──────────────┘
   ※ 休憩タイマー自動開始、完了時に触覚フィードバック（.notification）

5. 音声入力対応
   「ベンチプレス 82.5キロ 8回」→ パース → 確認 → 記録

6. ワークアウト完了
   ┌──────────────┐
   │ 🏋️ 完了！      │
   │ 45分 / 12セット │
   │ 総挙上量 2,880kg│
   │              │
   │ BULK:         │
   │ 「今日もナイス！│
   │  来週は85kg   │
   │  狙ってみる？」│
   └──────────────┘
```

### iPhoneアプリ（コンパニオン）

Watchで記録、iPhoneで振り返る：

- **ダッシュボード**: 成長グラフ（挙上重量、推定1RM、ボリューム推移）
- **BULKとの会話**: フルチャットUI。ワークアウトの振り返り、次のメニュー相談
- **カレンダー**: ワークアウト履歴。ストリークは表示するが、切れても罰はない
- **PR記録**: 自己ベスト一覧。BULKのコメント付き
- **体の変化**: 写真記録（任意）。BULKが変化をコメント

## 技術アーキテクチャ

### スタック

```
┌──────────────────────────────────────────┐
│  Apple Watch App（主役）                    │
│  ├── SwiftUI                              │
│  ├── SwiftData（ローカルデータ）             │
│  ├── HealthKit（心拍・カロリー・ワークアウト）  │
│  ├── WorkoutKit（ワークアウトセッション）      │
│  └── WidgetKit（コンプリケーション）          │
├──────────────────────────────────────────┤
│  iPhone Companion App（ダッシュボード）       │
│  ├── SwiftUI                              │
│  ├── SwiftData（共有データモデル）            │
│  ├── Charts framework（成長グラフ）          │
│  └── WatchConnectivity（同期）             │
├──────────────────────────────────────────┤
│  Cloud Backend                            │
│  ├── CloudKit（無料、Apple標準同期）          │
│  ├── LLM API（Claude / GPT）              │
│  │   └── ストリーミング応答                  │
│  └── Edge Functions（Cloudflare Workers）  │
│      └── LLMプロキシ、プロンプト管理          │
└──────────────────────────────────────────┘
```

### 重要な設計判断

1. **HealthKitはセット/レップ/重量を保存できない** → SwiftDataでカスタム保存、HealthKitは心拍・カロリーのみ
2. **LLMはオフライン時に使えない** → コアの記録機能はLLM非依存。BULKの会話はオンライン時のみ
3. **Watch上のLLM呼び出し遅延は1-5秒** → セット間の短い一言はプリキャッシュ、長い会話はiPhone側で
4. **Native Swift必須** → React Native/Flutterはwatch OS非対応。Swift + SwiftUI一択

| フェーズ | 期間 | 内容 |
|---------|------|------|
| 設計・データモデル | 1-2週 | UI設計、HealthKitスキーマ |
| Watch コア（記録） | 3-4週 | ワークアウトUI、セットロギング、レストタイマー |
| HealthKit連携 | 1-2週 | ワークアウトセッション、心拍 |
| BULK AI連携 | 2-3週 | LLM API統合、プロンプト設計、キャラクター構築 |
| iPhone ダッシュボード | 2週 | 履歴、グラフ、BULKチャット |
| コンプリケーション | 1週 | 文字盤表示 |

| テスト・ポリッシュ | 2週 | 実機テスト |
| App Store申請 | 1週 | |
| **合計** | **14-18週** | **フルタイムで約4ヶ月** |

## 専門家評価

### 投資家（収益性・スケーラビリティ）: B+

**ポジティブ**:
- TAMは巨大。グローバルフィットネスアプリ市場$12B+、2033年$34-45B
- 「運動が続かない人」は市場の80%。最大のペイン
- AI companion × フィットネスの掛け合わせは未開拓。Replika（$24M ARR）× Duolingo（$1B+ ARR）の交差点
- サブスクモデルで予測可能な収益

**ネガティブ**:
- B2Cモバイルは最も厳しい市場。CAC高い、チャーン高い
- LLM APIコストがユーザーあたり月$1-3。低単価サブスクでマージンが薄い
- Apple Watch必須は市場を狭める
- 個人開発でB2Cモバイルは非常に困難

**評価**: 「アイデアは面白いが、個人開発でB2Cモバイルは茨の道。まずiPhoneのみで検証し、Watch対応はPMF後に。LLMコストを月$1以下に抑える設計が必須」

### マーケター（集客・バイラリティ・GTM）: A-

**ポジティブ**:
- BULKのキャラクターマーケティングは強い。SNSでBULKの面白い返しが拡散される
- 「筋トレが続かない」は共感度が高い。マーケティングメッセージが明確
- TikTok/YouTube Shorts/Xでの短尺コンテンツと相性抜群
- 創業者自身がペルソナ → ビルドインパブリックの説得力が高い

**GTM戦略**:
1. **Phase 0**: 創業者の筋トレ日記（X/TikTok）で0→1000フォロワー
2. **Phase 1**: BULKのキャラクターをSNSで先行展開。「BULKがこう言ってきた」スクショ共有
3. **Phase 2**: TestFlight限定ベータ（100人）→口コミ
4. **Phase 3**: App Storeローンチ + ProductHunt

**バイラルループ設計**:
- ワークアウト完了時にBULKが生成するサマリーカード → SNS共有
- PR更新時のBULKの反応 → スクショ共有
- 月間サマリー（「今月5トン持ち上げた」） → シェア

**ネガティブ**:
- フォロワー0からの集客は時間がかかる
- App Store検索で上位を取るのは困難（SEO競争過密）
- 日本市場だけでは規模が限定的。グローバル展開必須だが多言語LLMプロンプト管理が複雑

### 筋トレ専門家（トレーニング理論・実用性）: B

**ポジティブ**:
- プログレッシブオーバーロード（漸進的過負荷）の自動計算は本質的な価値
- Watch完結のセットロギングは実用的。セット間のスマホ離脱を防ぐ
- レストタイマーの触覚フィードバックは地味に重要（ジムは騒がしい）
- BULKの「今日は軽めにしよう」提案は、オーバートレーニング防止に有効

**改善要望**:
- **フォーム指導が必須**: 初心者が怪我するリスク。3Dアニメーション or 動画ガイドを入れるべき
- **RPE（主観的運動強度）入力**: 重量だけでなく「きつさ」を記録できると、プログラム最適化の精度が上がる
- **ウォームアップセット管理**: トレーニングの基本。無視してはいけない
- **デロード週の提案**: 4-6週ごとに強度を落とす週を自動提案すべき
- **エクササイズバリエーション**: 同じ種目の代替種目を提案できるとマンネリ防止になる

**ネガティブ**:
- 「ゆるく続ける」は初心者には正しいが、中級者以上はプログラムの科学的根拠を求める
- BULKの提案が運動科学的に正しくない場合、信頼を失う
- カロリー計算・栄養管理機能がないと、「筋トレ」だけでは成果が出にくい（ただしスコープ外で良い）

**評価**: 「コンセプトは面白い。ただし"ゆるさ"と"科学的正確性"のバランスが難しい。BULKのアドバイスに運動科学の裏付けがないと、ただの"ノリの良いAI"で終わる。フォーム指導機能は必須」

### レッドチーム（失敗シナリオ・致命的リスク）: 失敗確率 80%

**致命的リスク6つ**:

| # | リスク | 重症度 | 詳細 |
|---|--------|-------|------|
| 1 | **BULKの品質がプロダクトの全て** | 致命的 | LLMの応答品質が悪ければ即離脱。テンプレ感が出た瞬間に冷める。プロンプトエンジニアリングの難度が極めて高い |
| 2 | **LLM APIコスト** | 致命的 | 月980円のサブスクで、LLM API月$1-3/ユーザーかかるとマージンが出ない。会話量が増えるほど赤字 |
| 3 | **Apple/Googleが同機能を内製化** | 高 | Apple Fitness+にAIコーチが搭載される可能性。OS統合の優位性には勝てない |
| 4 | **B2Cモバイルのチャーン** | 高 | フィットネスアプリの30日後残存率は8-12%。BULKで改善しても30%が限界か |
| 5 | **Watch必須は市場を狭める** | 高 | Apple Watch保有率はiPhoneユーザーの30%程度。市場が1/3に |
| 6 | **個人開発でネイティブiOS + watchOS** | 高 | Swift経験がない場合、学習コスト+開発期間が想定の2倍に。副業で1年以上かかる |

**失敗シナリオ**:

1. **「BULKがつまらない」シナリオ（確率40%）**: LLMプロンプトの限界で、BULKの返しがパターン化。2-3週間で「このAI、同じことしか言わない」と冷める → チャーン
2. **「コスト構造破綻」シナリオ（確率30%）**: アクティブユーザーが増えるほどLLM APIコストが膨らみ、月980円では赤字 → 値上げ → チャーン
3. **「開発が終わらない」シナリオ（確率50%）**: 副業でネイティブiOS + watchOS開発は重い。1年経ってもリリースできず、モチベーション消失
4. **「Hevy/Strongの壁」シナリオ（確率30%）**: 記録機能の完成度でHevy/Strongに勝てず、BULKだけでは差別化不十分

**レッドチームの提案**:

> BulkTrackを**iPhoneアプリとして先にリリース**し、BULKの価値を検証すべき。Watch対応はPMF確認後の拡張。SwiftでiPhoneアプリを作り、BULKとの会話と基本的な記録機能だけでMVPを検証する。Watch対応は「ユーザーが求めてから」で十分。

### LLMコスト削減戦略（致命的リスク#2への対策）

1. **プリキャッシュ**: セット完了時の一言は、ユーザーの状況に基づいて50パターンをプリ生成。LLM呼び出し不要
2. **オンデバイス推論**: Apple CoreMLで小型モデルを動かし、簡単な応答はデバイス内で完結
3. **会話セッション制限**: Free: 1日1セッション、Pro: 1日5セッション、Pro+: 無制限
4. **バッチ処理**: ワークアウト後のサマリーは即時生成せず、数秒待ってバッチで処理
5. **コンテキスト圧縮**: 長い会話履歴をサマリー化し、トークン数を削減

## アイデアを尖らせる: 5つの「これがないと死ぬ」要素

リサーチと評価を統合した結果、BulkTrackが生き残るために**絶対に外してはいけない要素**:

### 1. BULKの「記憶」が本物であること

BULKがユーザーの全ワークアウト履歴を文脈として持ち、「先週の木曜のスクワットで膝が痛いって言ってたよね」レベルの記憶を見せる。これがテンプレ応答と本物のバディの分水嶺。

**実装**: ユーザーごとのコンテキストDB → LLMプロンプトにインジェクション

### 2. 「60点でいい」の設計思想

完璧主義が筋トレを殺す。BulkTrackは：
- ストリークが切れても罰なし
- 10分のワークアウトでも「やった」とカウント
- BULKは「今日はスクワット3セットだけでいいよ」と言える
- 週1回でも「継続」と定義する

### 3. Watch上の"3タップ完了"記録

ワークアウト記録が面倒な瞬間にアプリは死ぬ：
- タップ1: セット開始
- タップ2: 完了（Digital Crownで重量・レップ数微調整）
- タップ3: 次のセットor種目変更

### 4. BULKが生成するシェアカード

ワークアウト完了時に、BULKが自動で「シェアしたくなるカード」を生成：
```
┌─────────────────────────┐
│  🏋️ Today's Report       │
│  by BULK                 │
│                          │
│  ベンチプレス 85kg × 8 × 3│
│  スクワット 100kg × 5 × 5 │
│                          │
│  総挙上量: 4,540kg        │
│  (小型車1台分)            │
│                          │
│  BULKの一言:              │
│  「スクワット100kgの壁、   │
│    ついに超えたじゃん！」   │
│                          │
│  #BulkTrack              │
└─────────────────────────┘
```

これが最大のバイラルループになる。

### 5. "おかえり"の設計

1週間、1ヶ月、3ヶ月サボっても、BULKは「おかえり！今日は何やる？」だけ。
罪悪感ゼロ。これが競合との最大の差別化であり、「カフェの店員」の本質。

**あの店員は、1ヶ月ぶりに来ても「いらっしゃいませ」と笑顔で迎えてくれる。「最近来てなかったですね」とは絶対に言わない。**
